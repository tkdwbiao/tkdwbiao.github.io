面试官：一亿条数据，如何找出重复次数最多的前三名？

2026-03-19 18:00
阅读时长: 3 分钟

在算法面试中，当数据量从“几千个”上升到“一亿个”时，问题的本质就从“找逻辑”变成了“控资源”。今天阿彪带大家拆解一道经典的大数据面试题。

1. 题目背景

题目：给定一个包含一亿个数据的超大文件（假设是长字符串，如 URL），物理内存限制为 2GB。请找出其中重复次数最多的前三个数据。

痛点分析：

  • 内存溢出 (OOM)：一亿条数据如果直接存入 Python 的 dict，由于 Python 对象的内存开销，至少需要 10GB~20GB 的内存，远超 2GB 的限制。
  • 无法排序：直接对一亿条数据进行快排的时间复杂度是 $O(N \log N)$，且无法在内存中完成。

2. 核心算法思想：分而治之 (Divide and Conquer)

处理海量数据的标准三步走：分片（Partition） -> 统计（Count） -> 合并（Merge）。

第一步：哈希分片 —— 保证“相同即同组”

我们不能把一亿条数据全读进来，但可以“路过”它们。

  1. 取模分流：遍历大文件，对每一条数据计算 hash(data) % 100。
  2. 原理：由于哈希函数的稳定性，相同的数据计算出的结果必然相同。
  3. 结果：大文件被拆分成了 100 个小文件（part_0.txt 到 part_99.txt）。平均每个文件包含 100 万条数据。

“这一步的精髓：将‘全局内存不足’转化为‘局部内存可控’。”

第二步：局部统计 —— 逐个击破

现在我们手里有 100 个小箩筐，每个约 10MB~100MB，内存完全塞得下。

  1. 清空内存，依次读取每一个小文件。
  2. 哈希计数：使用 Python 的 collections.Counter 或 dict 进行词频统计。
  3. 记录前三：统计完当前文件后，只记录下该文件里的 Local Top 3。
  4. 释放内存，处理下一个文件。

第三步：全局聚合 —— 最小堆筛选

  1. 汇总：遍历完 100 个文件后，我们得到了 300 个候选者。
  2. 最终竞赛：维护一个大小为 3 的最小堆（Min-Heap）。
  3. 逻辑：
    • 遍历这 300 个候选者。
    • 如果当前数据的频次比堆顶（当前前三名里最菜的那个）大，就踢掉堆顶，把当前数据压入堆中。
  4. 产出：处理完毕后，堆里的 3 个数据就是最终答案。

3. 深度思考：如果分片不均怎么办？

这是阿彪在刷题时想到的一个硬核细节：数据倾斜 (Data Skew)。

  • 场景 A：数据重复度极高。比如某个分片里有 500 万条数据，但全是 “Apple”。
    • 阿彪结论：安全。因为哈希表只存储唯一键，{"Apple": 5000000} 只占用一行内存。
  • 场景 B：数据各不相同且分片不均。比如 part_0.txt 倒霉地分到了 5000 万个不同的字符串。
    • 阿彪对策：采用 “二次哈希”。如果发现某个分片依然超载，针对这个特定文件更换哈希种子再次拆分，直到内存可控。

4. 复杂度分析

  • 时间复杂度：$O(N)$。
    • 分片遍历一遍数据：$O(N)$。
    • 统计遍历一遍数据：$O(N)$。
    • 堆排序处理常数级数据：$O(1)$。
  • 空间复杂度：$O(M)$。
    • $M$ 是单个分片中唯一数据的规模，通过控制分片数量，可以将 $M$ 压制在物理内存以下。

5. 阿彪的面试金句 [抱拳]

  1. 资源意识：主动提出“内存只有 2GB”这个约束条件，展示你对底层资源的敏感度。
  2. 稳定性：强调 hash % N 保证了相同元素必进同组，这是逻辑成立的前提。
  3. 工具选型：在合并阶段提到“最小堆”，证明你具备处理 Top K 问题的标准工具箱。
  4. 架构思维：提及“二次哈希”处理数据倾斜，向面试官证明你考虑到了工业界的极端情况。

海量数据处理不难，核心就是“化整为零”。阿彪希望这篇文章能帮你建立起这种‘分治’的直觉！

二分查找终极奥义：从‘逻辑模糊’到‘一招制敌’的万能公式

2026-03-19 20:00
阅读时长: 5 分钟

二分查找（Binary Search）是算法面试中最容易“一看就会，一写就废”的题目。为什么？因为 left < right 还是 left <= right？返回值是 left 还是 right？这些细节能把人绕晕。

今天，阿彪带大家建立一套永远不会乱的心理模型：只写一种核心逻辑，通过 +1/-1 搞定所有变体。

1. 核心大招：死守 lower_bound

与其去记四五种不同的二分写法，不如只练熟一种最强大的：寻找第一个 $\ge$ Target 的位置。

def lower_bound(nums, target):
    # 使用闭区间 [left, right]
    left, right = 0, len(nums) - 1
    
    while left <= right:
        # 防止溢出的专业写法
        mid = left + (right - left) // 2
        
        if nums[mid] < target:
            # 目标在右边，左边界跳过 mid
            left = mid + 1
        else:
            # 目标在左边（或者就是 mid），右边界收缩
            right = mid - 1
            
    # 循环结束时，left 永远指向第一个满足 “>= target” 的位置
    return left

2. 万能转换公式：微调的艺术

阿彪总结出的精髓：无论题目要求找什么，都先调用 lower_bound，然后根据以下公式微调。

你想找的目标等价转换逻辑阿彪的代码写法
第一个 $\ge X$基础原型lower_bound(nums, X)
第一个 $> X$等于找 第一个 $\ge (X + 1)$lower_bound(nums, X + 1)
最后一个 $< X$第一个 $\ge X$ 的左边一个lower_bound(nums, X) - 1
最后一个 $\le X$第一个 $> X$ 的左边一个lower_bound(nums, X + 1) - 1

3. 为什么这个公式无敌？（深度拆解）

我们用例子 nums = [1, 2, 4, 4, 5]，X = 4 来套公式：

  1. 找第一个 $\ge 4$：
    • 调用 lower_bound(4)，返回索引 2（对应第一个 4）。完全正确。
  2. 找第一个 $> 4$：
    • 在整数世界里，大于 4 的第一个数就是 5。
    • 等价于找“第一个 $\ge 5$”。
    • 调用 lower_bound(5)，返回索引 4（对应 5）。完全正确。
  3. 找最后一个 $< 4$：
    • 小于 4 的最大数，一定就在“第一个 $\ge 4$”的左边。
    • 调用 lower_bound(4) - 1，即 2 - 1 = 1（对应 2）。完全正确。
  4. 找最后一个 $\le 4$：
    • 小于等于 4 的最大数，其实就是“第一个 $> 4$”的左边。
    • 调用 lower_bound(4 + 1) - 1，即 4 - 1 = 3（对应最后一个 4）。完全正确。

4. 实战演练：LeetCode 34 题

题目：在排序数组中查找元素的第一个和最后一个位置。 阿彪分析：

  • 第一个位置 = 第一个 $\ge$ target。
  • 最后一个位置 = 最后一个 $\le$ target $\rightarrow$ 转换为公式：第一个 $> target$ 的左边一个。

class Solution:
    def searchRange(self, nums: List[int], target: int) -> List[int]:
        # 直接复用 lower_bound 逻辑
        def lower_bound(t):
            l, r = 0, len(nums) - 1
            while l <= r:
                m = l + (r - l) // 2
                if nums[m] < t: l = m + 1
                else: r = m - 1
            return l

        # 1. 找左边界
        start = lower_bound(target)
        
        # 边界检查：如果没找到，或者找到的数不对
        if start == len(nums) or nums[start] != target:
            return [-1, -1]
            
        # 2. 找右边界：利用公式 lower_bound(target + 1) - 1
        end = lower_bound(target + 1) - 1
        
        return [start, end]

5. 阿彪悟道：为什么这个方法最科学？

① 关于逻辑归一化

很多同学在写二分时，一会儿写 if nums[mid] > target，一会儿写 if nums[mid] == target。 阿彪建议：永远只关注 “什么时候该向右走？” 即 nums[mid] < target 的时候。其他所有情况统一收缩右边界。这样逻辑极其纯粹。

② 关于“红蓝染色”思维

你可以想象：

  • 数组左边是不满足条件的“红色”区域。
  • 数组右边是满足条件的“蓝色”区域。
  • while left <= right 结束后，left 永远踩在蓝色的第一个格子上，right 永远踩在红色的最后一个格子上。

③ 关于边界溢出

  • 如果返回的 left == len(nums)：说明全场都比 Target 小，没找着。
  • 如果返回的 left == 0 且 nums[0] != target：说明全场都比 Target 大，也没找着。

6. 阿彪可能没想到的两个“坑”

当你用这个转换公式时，一定要注意越界检查：

  • 坑一：找不到怎么办？ 如果 lower_bound 返回的值等于 len(nums)，说明数组里所有的数都比你给的 target 小。
    • 面试话术：“我会检查返回值是否越界，以确定目标是否存在于数组区间内。”
  • 坑二：数组里没这个数怎么办？ 比如找“第一个 $\ge 3$”，但数组里是 [1, 2, 4, 5]。
    • lower_bound(3) 会返回索引 2（对应 4）。
    • 逻辑依然成立：4 确实是第一个大于等于 3 的数。

7. return left 和 return right + 1 有何差异：

直接给结论：在 while left <= right 循环结束的那一刻，left 和 right + 1 在数学上是完全相等的。

所以，写 return left 和写 return right + 1 效果一模一样。我来给你拆解一下这背后的“指针交错”逻辑。

① 为什么 right + 1 等于 left？

还记得我们之前说的“红蓝染色法”和“擦肩而过”吗？

  • 循环进行中：只要 left <= right，说明区间里还有数，大家继续找。
  • 循环结束时：最后一步一定是 left 和 right 停在同一个位置，然后其中一个动了一下，导致 left > right。

因为每次移动都是 +1 或 -1，所以当它们刚发生错位时，状态一定是：

left = right + 1

这就是为什么有的题解写 left，有的题解写 right + 1，它们只是在描述同一个位置的两种表达方式。

② 深度对比：left vs right + 1

虽然数值相等，但它们在脑海里的物理意义略有不同：

返回 left 的意义（阿彪心法）：

  • 直觉：left 代表“新世界的开头”。
  • 含义：数组中有多少个元素是比 target 小的？答案正好就是 left。
  • 插入点：如果你要把 target 插进去，这个索引位置就是 left。

返回 right + 1 的意义：

  • 直觉：right 停在了“旧世界的末尾”。
  • 含义：既然 right 是最后一个比 target 小的数，那插入位置理所应当就在它后面一个，即 right + 1。

③ 阿彪的“避坑”建议：选哪个更好？

我强烈建议你以后统一写 return left。

原因有三：

  1. 更省脑细胞：你不需要去算加减法，直接返回那个一直在“冲锋”的 left 指针。
  2. 语义更强：在绝大多数二分变体中，left 代表的是 “第一个满足条件的索引”，这已经成了算法界的潜规则。
  3. 防止越界直觉错误：写 right + 1 有时候会让你担心 +1 会不会超出数组长度，虽然在返回值这里没问题，但心理压力大。

8. 阿彪的面试金句 [抱拳]

“面试官，二分查找的精髓在于对单调区间的二段划分。我习惯于将所有的查找变体（如找大于、找小于等于）统一收缩到 lower_bound 逻辑上。通过对 Target 值或 Result 索引进行 ±1 的修正，可以极大地减少边界条件的判断，保证代码的鲁棒性。”

二分查找的 while 循环是一场指针的赛跑，当 left 与 right 擦肩而过，它们共同定义了一个分界点。right 守住了旧世界的最后防线，而 left 踏上了新世界的第一块基石。 无论你返回 left 还是 right + 1，你拿到的都是通往新世界的钥匙。

掌握了 +1/-1 的微调魔法，二分查找从此不再有坑！
